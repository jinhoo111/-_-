# 01. 아이디에이션 가이드

> **이 파일이 필요한 시점:** 유저가 사업 아이템·아이디어 도출을 요청할 때.

-----

## 0. 기본 사고 프레임

이 프로젝트의 아이디에이션은 다음 3가지 사고 프레임을 결합해서 진행합니다.

### A. 컨셉 빌딩 (Concept Building) — 『끌리는 컨셉 만들기』 김근배, 중앙북스, 2018

**핵심 명제: “바라는 결과(문제정의)” 와 “충족수단(해결책)” 을 분리해서 사고하라.**

- 컨셉 = 다른 제품이 아닌 **이 제품을 사야 할 이유** 를 제시하는 것
- 사야 할 이유 = 고객가치 = 제품/서비스가 고객이 바라는 결과를 제공하는 것
- 창의적 사고 = 지성(개념화) × 공감 × 상상 × [왜?]
- 컨셉 빌딩 6층 구조 (지하 = 회사 내부 / 지상 = 외부 노출):
  - B3F **Prototype** (손으로 상상)
  - B2F **Ideation** (머리로 상상)
  - B1F **Concept** (개념화 / 지성)
  - 1F **Sympathy** (고객공감)
  - 2F **Metrics** (컨셉 수익성 계량지표)
  - 3F **Business Plan** (사업기획)
- 12 도구 (4개 × 3 영역):
  - 고객공감: 질문 / 관찰 / 추체험 / 고객 피드백
  - 개념화(지성): 미충족 니즈 발견표 / 컨셉 재정의표 / 핵심편익 정의·컨셉 서술문 / 컨셉보드
  - 상상력: 유추 / 가추(가정추론) / 변증추론으로 모순해결 / 공감적 상상

> 출처: 김근배, 『끌리는 컨셉 만들기』, 중앙북스, 2018. (이미지 첨부본 기반)

### B. Jobs-to-be-Done (JTBD)

“고객은 제품을 사는 게 아니라 어떤 **‘일(job)’** 을 끝내기 위해 제품을 **고용한다**” 는 관점.

- Job Story 포맷: **”[상황]에서, [동기]를 원해서, [원하는 결과]를 얻고 싶다.”**
  예) “출퇴근 지하철에서, 지루함을 잊고 싶어서, 짧은 시간 안에 즐길 콘텐츠를 얻고 싶다.”
- 좋은 Job Statement는 **솔루션-비의존적**입니다. 즉 “이 앱을 쓴다”가 아니라 “이런 결과를 얻는다”로 표현.
- 현재 유저가 **무엇을 hire / fire 하고 있는지** 를 봅니다. 그 갭이 기회.

> 출처: Tony Ulwick (Strategyn), Outcome-Driven Innovation https://strategyn.com/jobs-to-be-done/ ; First Round Review (Sunita Mohanty), “Build Products That Solve Real Problems” https://review.firstround.com/build-products-that-solve-real-problems-with-this-lightweight-jtbd-framework/

### C. “1-2주 안에 안드로이드 앱으로 출시 가능한가?” 필터

이 프로젝트의 핵심 제약입니다. 모든 아이디어는 다음 질문을 통과해야 합니다.

- 클로드 코드로 1-2주 안에 MVP 가능한가? (복잡한 ML, 외부 하드웨어 의존 X)
- 안드로이드 단일 플랫폼으로 검증 가능한가?
- 핵심 가치를 5분 만에 사용자가 체험 가능한가?

-----

## 1. 입력 질문 체크리스트

유저가 그냥 “사업 아이템 뽑아줘” 라고 하면 다음 항목들을 빠르게 묻습니다. 한 번에 다 묻지 말고, 핵심 3-4개만 먼저 묻고 나머지는 답변에 따라 후속 질문하세요.

|#|질문 항목                                |비고            |
|-|-------------------------------------|--------------|
|1|**사업 아이디어 소개** (있는 거 한 줄)            |아예 없으면 ‘없음’   |
|2|**아이디어 떠올린 배경** (데스크리서치 자료 / 본인 경험 등)|출처 있으면 링크     |
|3|**사업 아이템의 도메인**                      |아래 리스트 중 택    |
|4|**사업 아이템의 시장**                       |도메인보다 구체적인 시장 |
|5|**메인 타겟**                            |페르소나 한 줄      |
|6|**서브 타겟** (있다면)                      |              |
|7|**그들이 겪는 문제**                        |              |
|8|**문제의 배경**                           |왜 이 문제가 지금 발생?|
|9|**기타 얘기하고픈 것**                       |자유            |

### 도메인 리스트 (모두의 창업 분류 기반)

교육 / 금융 / 운영관리 / 네트워크 / 농수산 / 마케팅 / 라이프스타일 / 모빌리티 / 미디어 / 바이오 / 에너지 / 유통 / 임팩트 / 재무 / 프롭테크 / 하드웨어 / IT

### 답변이 비어있을 때의 자동 보완 규칙

- **항목 1, 2, 7, 8 비어있음** → 검색 + 문맥으로 채워서 진행
- **모든 항목 비어있음** → 다음 3가지 관점으로 아이디어 뽑기:
1. **가장 빠르게 실현 가능한 것** (1주 안에 클로드 코드로 만들 수 있는 것)
1. **가장 참신한 것** (시장에 비슷한 게 거의 없는 것)
1. **가장 떠오르는 도메인에서의 아이디어** (현재 트렌드가 강한 도메인)

## 2. 아이디어 출력 템플릿

각 아이디어는 다음 8개 섹션을 **모두** 담아야 합니다. 최소 3개 버전 제시.

### 출력 포맷 (한 아이디어 단위)

```markdown
## 아이디어 [N]: [한줄로 표현한 이름]

### 1) 한줄 소개
[이게 뭐 하는 건지 한 문장으로]

### 2) 아이디어 배경
[왜 이게 필요한지 / 어떤 데스크리서치/관찰에서 나왔는지]
- 근거 데이터: [출처 링크 또는 (명확하지 않은 출처)]

### 3) 누구의 어떤 문제를 해결하는가
- **메인 타겟**: [구체적 페르소나]
- **JTBD (Job Story)**: "[상황]에서, [동기]를 원해서, [원하는 결과]를 얻고 싶다."
- **현재 그들이 hire 하는 것 / fire 하는 것**: [현재 대안과 그 한계]

### 4) 구체적 설명과 차별점
[기능 설명. 단, 기능 백화점 X. 킬러 기능 1개 + 보조 2-3개]
- **차별점**: [경쟁 대비 어떻게 다른가 — 1-2줄]

### 5) 어떻게 실현하면 좋을지
- **MVP 핵심 화면 3-5개**: [화면 이름 나열]
- **킬러 기능 (필수)**: [1개]
- **보조 기능**: [2-3개]
- **MVP에서 빼야 하는 것**: [욕심내고 싶지만 v1에서는 제외할 것]
- **예상 개발 기간**: [클로드 코드 기준 대략적 추정]

### 6) 도메인 / TAM / SAM / SOM
- **도메인**: [위 도메인 리스트 중 1-2개]
- **TAM** (전체 시장): [숫자 + 출처]
- **SAM** (서비스 가능 시장): [숫자 + 추정 근거]
- **SOM** (실제 획득 가능 시장, 1-2년 내): [숫자 + 추정 근거]
> TAM/SAM/SOM 계산 방식은 본 가이드 § 4 참고

### 7) 참고 레퍼런스 검색 키워드
- 한글: [키워드 3-5개]
- 영어: [키워드 3-5개]
- 추천 사이트: [Producthunt 카테고리 / App Store 카테고리 / 특정 블로그]

### 8) 경쟁사
- **직접 경쟁**: [동일/유사 가치 제안하는 서비스 2-3개, 한 줄 평]
- **간접 경쟁**: [같은 job을 다른 방식으로 해결하는 서비스/방법 2-3개]
```

### 마지막 요약 (3개 아이디어 비교)

응답 끝에 반드시 다음 비교표를 붙입니다.

```markdown

| # | 아이디어 | 메인 타겟 | 킬러 기능 | 도메인 | 강점 | 리스크 |
|---|---|---|---|---|---|---|
| 1 | ... | ... | ... | ... | ... | ... |
| 2 | ... | ... | ... | ... | ... | ... |
| 3 | ... | ... | ... | ... | ... | ... |
```

그리고 **“어떤 아이디어로 진행하시겠어요? 선택하시면 상세 정보를 `idea_[영어이름].md` 파일로 정리해드립니다.”** 라고 마지막에 묻기.

## 3. 아이디어 검증 체크리스트

각 아이디어를 출력하기 전 머릿속(혹은 답변 내부)으로 다음을 점검합니다.

- [ ] **솔루션이 아니라 Job으로 설명되는가?** “이 앱은 ~~한다” 가 아니라 “이 사용자가 ~~를 끝낼 수 있다” 로 표현되는가
- [ ] **차별점이 1줄로 설명되는가?** 안 되면 컨셉이 흐릿한 것
- [ ] **킬러 기능 1개를 식별할 수 있는가?**
- [ ] **1-2주 내 안드로이드 MVP가 가능한가?** (특수 하드웨어/대규모 ML 의존 X)
- [ ] **TAM이 최소 100억 원 이상인가?** (너무 작으면 의미 없음)
- [ ] **현재 hire/fire 패턴이 명확한가?** (사용자가 지금 뭘 쓰고 있는지 안다는 뜻)
- [ ] **컨셉 빌딩의 “왜 사야 하는가?” 에 답할 수 있는가?**

## 4. TAM / SAM / SOM 계산 가이드

### 정의

- **TAM** (Total Addressable Market): 만약 우리가 시장 100% 점유하면 얻는 매출
- **SAM** (Serviceable Available Market): 우리 비즈니스 모델/지역/언어로 도달 가능한 부분
- **SOM** (Serviceable Obtainable Market): 1-2년 내 현실적으로 획득할 수 있는 부분

### 추천 계산 방식: Bottom-up

시장조사 보고서 숫자를 그대로 쓰는 Top-down은 부풀려질 위험이 큽니다. 가능하면 Bottom-up으로:

```
TAM = (잠재 사용자 수) × (연간 평균 지불 금액)
SAM = TAM × (우리가 실제 닿을 수 있는 비율, 보통 10-30%)
SOM = SAM × (1-2년 내 점유 가능 비율, 보통 1-5%)
```

### 출처 신뢰 우선순위

1. 통계청, KOSIS (한국)
1. Statista, 한국정보통신정책연구원(KISDI), DMC Report
1. 업계 1위 기업의 IR/공시 자료
1. 논문/리포트
1. 블로그/언론 (가장 낮은 신뢰)

검증 가능한 링크는 web_fetch로 페이지 유효성 확인 후 인용. 그렇지 않으면 `(명확하지 않은 출처)` 표기.

## 5. 선택 후 출력: 상세 아이디어 MD 파일

유저가 한 아이디어를 선택하면, 위 섹션 1)~8) 을 그대로 담은 `idea_[영어슬러그].md` 파일을 생성합니다. 다음 항목을 추가로 포함:

### 추가 섹션

```markdown
### 9) 사용자 시나리오 (3개)
1. **첫 사용**: [신규 사용자가 앱을 처음 켜고 가치를 체감하는 시나리오]
2. **재방문**: [재방문하는 핵심 트리거와 동선]
3. **공유/입소문**: [어떤 순간에 다른 사람에게 공유하게 되는지]

### 10) 수익 모델 (1차 가설)
- **핵심 모델**: [광고 / 구독 / 인앱 결제 / 수수료 / etc.]
- **시점**: [언제부터 수익화 시작할지 — DAU 몇 명부터 등]
- **객단가 가설**: [얼마 정도 받을 수 있을지의 추정]

- **North Star Metric**: [이 앱의 본질적 성공을 측정하는 1개 지표]
- **Input Metrics**: [North Star를 움직이는 선행 지표 3-4개]

### 12) 리스크 & 가설 (검증 필요)
- **가장 큰 리스크 1개**: [무엇이 실패시킬 수 있는가]
- **검증 우선순위 가설 3개**:
  1. [가설] — 검증 방법: [방법]
  2. ...
  3. ...
```

## 6. 자주 빠지는 함정

이 함정에 빠진 아이디어는 출력에서 빼거나 수정해서 출력하세요.

1. **솔루션부터 시작한 아이디어**: “AI로 ~를 만든다” → 이건 솔루션. “어떤 사용자의 어떤 문제를 푸는가?” 가 먼저.
1. **너무 큰 비전**: “전국민이 쓰는 ~“ → MVP 단계에서는 좁고 뾰족해야 함.
1. **차별점이 “더 좋다” 류**: “기존보다 더 빠르고 예쁨” → 차별점은 **다르게 정의된 문제** 에서 나와야 함.
1. **타겟이 “20-30대”**: 이건 타겟이 아님. “주 3회 이상 자취 요리하는 1인 가구 30대 직장인” 정도까지 좁혀야 함.
1. **무료라서 쓴다**: 무료여서 쓰는 건 가치가 아님. 무료여도 다운로드만 받고 안 쓰는 앱이 99%.
1. **본인이 안 쓸 앱**: 개발자/디자이너 본인이 한 달에 5번 이상 쓸 자신이 없으면 보통 실패.
